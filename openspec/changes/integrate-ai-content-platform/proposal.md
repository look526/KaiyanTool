# 提案：AI内容创作平台整合

## 1. 背景与动机

当前项目包含三个独立的AI内容创作应用：

1. **BigBanana-AI-Director**：基于React的Web应用，专注于AI导演工作流，采用"Script-to-Asset-to-Keyframe"模式
2. **Toonflow-app**：基于Node.js + Express的后端应用，实现小说到视频的自动化工作流，采用多Agent架构
3. **CineGen-AI**：原型项目，使用Google Gemini 2.5 Flash和Veo模型

这三个项目功能互补但架构独立，缺乏统一的数据和用户体验。本提案旨在整合这三个项目的核心功能，构建一个综合性AI内容创作平台。

## 2. 项目详细分析

### 2.1 BigBanana-AI-Director

#### 核心功能清单
- **剧本解析**：将原始剧本文本结构化为可操作的数据格式（场景、角色、动作、对话）
- **角色资产管理**：
  - 支持多角色定义（姓名、年龄、性别、外貌描述）
  - 角色参考图像（Base64编码）
  - 角色服装变体（支持同一角色不同造型）
- **场景管理**：
  - 场景信息（地点、时间、氛围）
  - 视觉风格选择（live-action/2d-animation/3d-animation）
- **镜头列表生成**：基于剧本自动生成镜头序列
- **关键帧系统**：
  - 起始帧和结束帧提示词生成
  - 双帧优化（优化两个关键帧的视觉连贯性）
  - 镜头拆分为子镜头
- **九宫格预览系统**：
  - 生成9个关键帧面板
  - 单次生成九宫格图像
  - 可视化镜头规划
- **图像生成**：
  - 支持参考图像
  - 多种宽高比（16:9, 9:16, 1:1）
  - 图像变体生成
- **视频生成**：
  - 基于关键帧生成视频
  - 首尾帧连接生成
  - 多模型支持（Veo, Sora-2）
  - 时长控制（4s, 8s, 16s）

#### 技术栈
- **前端**：React 19.2.0 + TypeScript 5.8.2 + Vite 6.2.0
- **存储**：IndexedDB（客户端浏览器存储）
- **AI服务**：
  - 主提供商：AntSK API (api.antsk.cn)
  - 支持模型：GPT-5.1, GPT-5.2, Gemini 3 Pro, GLM-4
  - 视频模型：Veo 3.1, Sora-2
- **状态管理**：React Hooks + Context API
- **UI组件**：自定义组件系统

#### 数据流程
```
用户输入剧本 → parseScriptToData → 结构化剧本数据
    ↓
generateShotList → 镜头列表
    ↓
用户定义角色/场景 → generateAssets → 角色和场景图像
    ↓
optimizeBothKeyframes → 起始/结束关键帧提示词
    ↓
generateImage → 关键帧图像
    ↓
generateVideo → 最终视频
```

#### 核心优势
1. **纯Web应用**：无需安装，浏览器即可使用
2. **关键帧驱动**：精确控制视频起止状态
3. **九宫格系统**：高效的镜头规划工具
4. **角色一致性**：通过参考图像保持角色外观一致性
5. **模型灵活性**：支持多种AI模型提供商

#### 应用场景
- 短视频创作者快速制作视频
- 广告分镜和预览
- 教育视频制作
- 概念视频验证

#### 潜在问题
- IndexedDB存储容量有限（浏览器限制）
- 无用户系统和云端同步
- 无协作功能
- 数据本地存储，换设备无法访问

### 2.2 Toonflow-app

#### 核心功能清单
- **小说管理**：
  - 小说基本信息管理
  - 多章节支持
- **故事线生成**：
  - AI1 Agent：分析小说生成故事线
  - 故事线保存和管理
- **大纲生成**：
  - AI2 Agent：基于故事线生成详细大纲
  - 多集支持（默认10集）
  - 大纲编辑、更新、删除
- **剧本生成**：
  - Director Agent：将大纲转换为剧本
  - 镜头级对话和动作描述
  - 剧本存储和管理
- **资产管理**：
  - AI自动生成角色和场景资产
  - 资产与剧本关联
  - 资产分类管理
- **分镜板生成**：
  - Segment Agent：将剧本分段
  - Shot Agent：为每个镜头生成提示词
  - 九宫格图像生成
  - 图像分割为4x4网格
  - OSS云存储集成
- **数据库管理**：
  - SQLite持久化存储
  - Knex ORM封装
  - 数据完整性约束

#### 技术栈
- **后端**：Express 5.2.1
- **数据库**：SQLite + Knex ORM
- **AI框架**：LangChain
- **前端**：React
- **运行时**：Node.js 23.11.1+
- **云存储**：阿里云OSS

#### 数据流程
```
上传小说 → AI1 Agent → 故事线
    ↓
AI2 Agent → 详细大纲
    ↓
Director Agent → 剧本
    ↓
generateAssets → 角色和场景资产
    ↓
Segment Agent → 剧本分段
    ↓
Shot Agent → 镜头提示词
    ↓
generateShotImage → 九宫格图像
    ↓
图像分割 → 4x4网格 → OSS存储
```

#### 核心优势
1. **多Agent协作**：专业分工的AI Agent系统
2. **完整工作流**：从小说到分镜板的端到端流程
3. **后端服务**：Express提供稳定的API服务
4. **数据库持久化**：SQLite支持大量数据存储
5. **云存储集成**：OSS支持大文件存储
6. **数据库完整性**：外键约束保证数据一致性

#### 应用场景
- 小说作者快速将作品可视化
- 动画工作室前期制作
- 漫画分镜规划
- 电视剧/动画前期制作

#### 潜在问题
- 需要部署后端服务
- 需要服务器资源
- Agent系统复杂，维护成本高

### 2.3 CineGen-AI

#### 核心功能清单
- **剧本输入**：用户输入原始剧本
- **剧本结构化**：将剧本解析为场景、角色、对话
- **场景分割**：将剧本分割为可管理的场景片段
- **资产生成**：为角色和场景生成图像
- **关键帧生成**：为每个镜头生成起始和结束帧
- **视频生成**：基于关键帧生成视频片段
- **视频拼接**：将多个视频片段拼接成完整视频

#### 技术栈
- **前端**：React + TypeScript
- **AI模型**：Google Gemini 2.5 Flash（文本）、Veo（视频）
- **视频处理**：FFmpeg
- **状态管理**：React Context

#### 数据流程
```
剧本输入 → 结构化解析 → 场景分割
    ↓
资产生成（角色、场景）
    ↓
关键帧生成
    ↓
视频生成
    ↓
视频拼接 → 最终输出
```

#### 核心优势
1. **简单直观**：四阶段线性工作流
2. **Google生态**：深度集成Google AI模型
3. **轻量级**：原型性质，代码简洁

#### 应用场景
- 概念验证
- 快速原型开发
- 教学演示

#### 潜在问题
- 功能相对基础
- 无角色一致性系统
- 无九宫格预览
- 无多模型支持
- 未完全实现

## 3. 功能整合方案

### 3.1 整合目标

构建一个统一的AI内容创作平台，具备以下特性：

1. **Web应用**：基于React的纯Web应用，无需安装，跨浏览器支持
2. **统一数据模型**：云端数据库 + 本地缓存
3. **完整工作流**：从剧本/小说 → 资产 → 关键帧 → 视频 → 最终输出
4. **多模式输入**：
   - 剧本模式（BigBanana风格）
   - 小说模式（Toonflow风格）
   - 混合模式
5. **智能Agent系统**：
   - 故事线Agent（AI1）
   - 大纲Agent（AI2）
   - 导演Agent（Director）
   - 分镜Agent（Segment/Shot）
6. **角色一致性系统**：
   - 角色参考图像
   - 服装变体管理
   - 角色资产库
7. **高级可视化**：
   - 九宫格预览
   - 关键帧编辑器
   - 分镜板视图
8. **云端协作**：
   - 项目共享
   - 多人协作
   - 版本控制

### 3.5 整合难点分析

#### 1. 数据模型差异

**问题**：
- BigBanana使用IndexedDB（本地、非结构化）
- Toonflow使用SQLite（本地、关系型）
- CineGen使用React Context（内存）

**解决方案**：
- 设计统一的云端数据模型（PostgreSQL）
- 前端实现离线缓存（IndexedDB）
- 数据同步机制（本地→云端→其他设备）

#### 2. AI Agent系统整合

**问题**：
- BigBanana无Agent系统，直接调用AI API
- Toonflow有完整的多Agent系统
- Agent逻辑分散在不同服务中

**解决方案**：
- 统一Agent抽象层
- 将Toonflow的Agent逻辑提取为独立服务
- BigBanana的功能作为Agent的补充工具

#### 3. 存储系统整合

**问题**：
- BigBanana使用Base64内联存储（性能差）
- Toonflow使用OSS（性能好）
- 大文件存储策略不统一

**解决方案**：
- 统一使用OSS/S3存储大文件
- 数据库只存储URL和元数据
- 前端缓存常用资产

#### 4. 用户体验一致性

**问题**：
- 三个应用的UI/UX风格不同
- 工作流程差异大
- 学习曲线陡峭

**解决方案**：
- 设计统一的UI/UX规范
- 提供模式切换（简单模式/高级模式）
- 渐进式功能引导

#### 5. 性能优化

**问题**：
- AI调用耗时长
- 大文件传输慢
- 实时预览要求高

**解决方案**：
- 异步任务队列
- 智能缓存策略
- 渐进式加载
- Web Worker处理耗时任务
